# 一、项目总体定位

项目名称可以定为：

> **基于 RBAC 的高校学生信息动态脱敏系统**

或者：

> **面向高校学生信息安全保护的动态数据脱敏与访问审计系统**

项目场景是：

> 高校学生信息系统中存储了大量学生个人敏感数据，例如姓名、学号、手机号、邮箱、身份证号、家庭住址、成绩、家庭收入、银行卡号等。不同用户在教学管理、数据分析、安全管理和系统维护过程中，对数据的访问需求不同。为了防止敏感数据被过度暴露，系统基于 RBAC 权限模型和字段级脱敏规则，在查询结果返回前动态完成数据脱敏，并记录敏感数据访问日志。

------

# 二、总体业务目标

系统要解决的问题是：

```text
不同角色访问同一份学生数据时，看到的数据不一样。
```

例如同一条学生记录：

```text
姓名：张三
手机号：13812345678
身份证：510104200501012345
地址：四川省成都市武侯区一环路南一段24号
GPA：3.82
家庭收入：86000
银行卡号：6222021234567890123
```

不同角色看到：

| 角色       | 展示效果                             |
| ---------- | ------------------------------------ |
| 超级管理员 | 完整原始数据                         |
| 教师       | 教学相关字段可见，高敏感字段轻度脱敏 |
| 数据分析师 | 身份类字段脱敏，数值类字段泛化       |
| 普通用户   | 高度脱敏数据                         |
| 安全审计员 | 不看业务数据，只看访问日志           |
| 安全管理员 | 配置规则，不直接查看业务原始数据     |
| 数据管理员 | 管理学生基础数据，一般查看脱敏数据   |
| 系统管理员 | 管理用户角色权限，不接触业务数据     |

# 三、最终角色设计

目前采用 8 类角色。

| 角色编码         | 角色名称      | 核心定位                             |
| ---------------- | ------------- | ------------------------------------ |
| `SUPER_ADMIN`    | 超级管理员    | 拥有全部权限，用于系统最高管理       |
| `SYSTEM_ADMIN`   | 系统管理员    | 管用户、角色、权限，不负责业务数据   |
| `DATA_ADMIN`     | 数据管理员    | 管学生数据资源、字段、数据质量       |
| `SECURITY_ADMIN` | 安全管理员    | 管敏感字段、脱敏规则、安全策略       |
| `TEACHER`        | 教师/教务人员 | 查询学生信息，用于教学和教务管理     |
| `ANALYST`        | 数据分析师    | 使用脱敏/泛化数据做统计分析          |
| `AUDITOR`        | 安全审计员    | 查看访问日志、规则变更日志和异常访问 |
| `NORMAL`         | 普通用户      | 低权限用户，只能查看高度脱敏数据     |

其中最容易混淆的四个角色可以这样记：

```text
数据管理员：管数据
数据分析师：用数据
安全管理员：配规则
安全审计员：查行为
```

# 五、模块一：登录认证模块

## 1. 功能目标

用户通过账号密码登录系统，系统识别用户身份、角色和权限。

## 2. 功能内容

| 功能         | 说明                               |
| ------------ | ---------------------------------- |
| 用户登录     | 输入用户名和密码                   |
| 密码校验     | 后端校验密码哈希                   |
| Token 生成   | 登录成功后返回 JWT 或 Session      |
| 角色加载     | 查询用户绑定的角色                 |
| 权限加载     | 查询角色拥有的菜单、接口、数据权限 |
| 前端菜单生成 | 根据菜单权限动态渲染页面           |

## 3. 登录后返回内容示例

```json
{
  "userId": 5,
  "username": "teacher01",
  "realName": "张老师",
  "roles": ["TEACHER"],
  "permissions": [
    "MENU:DASHBOARD",
    "MENU:STUDENT_QUERY",
    "API:STUDENT_QUERY",
    "API:STUDENT_DETAIL",
    "DATA:VIEW_MASKED",
    "DATA:QUERY_ALL_STUDENTS",
    "DATA:QUERY_SENSITIVE"
  ]
}
```

## 4. 登录后的业务逻辑

```text
1. 用户输入账号密码
2. 后端校验用户是否存在
3. 校验密码是否正确
4. 查询用户角色
5. 查询角色权限
6. 生成 Token
7. 前端根据菜单权限显示不同页面
```

# 八、模块四：学生信息查询模块

这是动态脱敏系统的核心展示模块。

## 1. 功能目标

不同角色查询学生信息时，根据权限和脱敏规则返回不同结果。

## 2. 主要角色

```text
SUPER_ADMIN
DATA_ADMIN
TEACHER
ANALYST
NORMAL
```

安全管理员和审计员一般不直接查询业务数据。

## 3. 查询方式

可以设计两种方式。

### 方式一：条件查询，更适合课程项目

前端提供筛选条件：

```text
学院
专业
年级
班级
姓名
学号
```

用户点击查询，后端拼接安全 SQL 查询。

优点是安全、稳定、易展示。

### 方式二：SQL 查询框，更贴近原开源项目

前端提供 SQL 输入框：

```sql
SELECT * FROM student_info;
```

优点是能体现数据库课程特点。

缺点是要处理 SQL 安全问题。

## 4. 建议采用方式

我建议你采用：

> **普通条件查询为主，SQL 查询框作为高级查询功能。**

这样更稳。

## 5. 查询业务流程

```text
1. 用户进入学生信息查询页面
2. 前端检查是否有 MENU:STUDENT_QUERY
3. 用户输入查询条件
4. 后端检查是否有 API:STUDENT_QUERY
5. 后端检查数据权限
6. 查询 student_info 原始数据
7. 判断用户是否有 DATA:VIEW_RAW
8. 如果有，返回原始数据
9. 如果没有，但有 DATA:VIEW_MASKED，则加载该角色脱敏规则
10. 对结果逐字段脱敏
11. 返回脱敏结果
12. 写入访问审计日志
```

## 4. 脱敏规则表

```sql
CREATE TABLE masking_rule (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '规则ID',

    table_name VARCHAR(100) NOT NULL COMMENT '表名',
    column_name VARCHAR(100) NOT NULL COMMENT '字段名',

    role_code VARCHAR(50) NOT NULL COMMENT '适用角色',

    sensitive_type VARCHAR(50) NOT NULL COMMENT '敏感类型',

    params JSON COMMENT '脱敏参数',

    enabled TINYINT DEFAULT 1 COMMENT '是否启用',

    create_by BIGINT COMMENT '创建人ID',
    update_by BIGINT COMMENT '更新人ID',

    create_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
```

## 1. 功能目标

| ------------ | ---------------------------------------- |
| 访问日志     | 记录用户查询了哪些表和字段               |

| 规则变更日志 | 记录谁新增、修改、删除了脱敏规则         |

## 4. 访问日志表


## 5. 规则变更日志表

# 十四、核心业务流程一：动态脱敏查询流程

```text
用户登录
  ↓
系统加载角色和权限
  ↓
用户进入学生查询页面
  ↓
后端校验 API 权限
  ↓
后端校验 DATA 权限
  ↓
执行数据库查询，得到原始结果
  ↓
识别结果字段中的敏感字段
  ↓
判断用户是否有 DATA:VIEW_RAW
  ↓
如果有，返回原始数据
  ↓
如果没有，加载该角色对应脱敏规则
  ↓
逐字段执行脱敏算法
  ↓
返回脱敏后的结果
  ↓
写入访问日志和脱敏命中日志
```

伪代码：

```java
if (!hasPermission(user, "API:STUDENT_QUERY")) {
    throw new AccessDeniedException("无权查询学生信息");
}

if (!hasPermission(user, "DATA:QUERY_ALL_STUDENTS")) {
    throw new AccessDeniedException("无权查询全部学生");
}

List<Student> rawData = studentService.query(condition);

if (hasPermission(user, "DATA:VIEW_RAW")) {
    auditLogService.record(user, condition, rawData, false);
    return rawData;
}

if (hasPermission(user, "DATA:VIEW_MASKED")) {
    List<Student> maskedData = maskingService.mask(rawData, user.getRoleCode());
    auditLogService.record(user, condition, maskedData, true);
    return maskedData;
}

throw new AccessDeniedException("无权查看数据");
```

# 二十一、最终项目逻辑总结

你的系统现在可以概括为：

> 系统基于 RBAC 模型设计用户、角色和权限管理，将权限划分为菜单权限、接口权限和数据权限。
> 在高校学生信息场景中，系统通过不同角色控制用户能访问哪些页面、能调用哪些接口、能查看什么粒度的数据。
> 当用户查询学生信息时，系统根据用户角色和脱敏规则，对姓名、手机号、身份证、地址、家庭收入、银行卡号等敏感字段进行动态脱敏。
> 同时，系统记录访问日志、脱敏命中日志和规则变更日志，实现学生敏感信息的最小化展示、动态保护和安全审计。

最终核心能力是：

```text
角色权限控制
字段敏感识别
动态脱敏处理
规则动态配置
查询结果可视化
访问日志审计
```

这就是目前完整方案。
